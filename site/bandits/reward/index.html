09 · Reward Loop

How live performance rewires the bandits

Every generated video that hits the ≥1,000 views floor feeds back into a Beta posterior update. This section is the math.

1
Select arms
At job create, each of 8 primary attributes samples from its Beta posterior.
θv ~ Beta(αv, βv)
arm = argmaxv θv
2
Record generation
Selected arm values written to dna_generation_configs alongside profile_date snapshot.
INSERT INTO dna_generation_configs
(job_id, hook_type_selected, …, profile_date)
3
Ingest feedback
Every 6h the TikTok Business API sync writes live GMV/views into dna_performance_feedback.
views ≥ 1000 · signal floor
bandit_updated = FALSE · work queue
4
Map GPM → reward
Binary reward vs. category baseline GPM at the time of generation (not now — avoids drift bias).
r = 1 if gpm ≥ P50category, profile_date
r = 0 otherwise
5
Update posterior
Conjugate update: one success bumps α, one failure bumps β. Cheap, exact, online.
αv′ = αv + r
βv′ = βv + (1 − r)
6
Persist & close loop
Upsert α/β into dna_bandits, mark row processed, next sample uses new posterior.
UPSERT dna_bandits (attribute, category, arm, α, β)
UPDATE dna_performance_feedback SET bandit_updated=TRUE

Signal floor
views ≥ 1,000 before any row counts. Sub-threshold rows stay unprocessed indefinitely.
Category floor
n < 50 videos → blended Bayesian prior from related verticals before any update.
Temporal baseline
Reward uses P50 at profile_date of the job, not today's P50. Drift can't retroactively reclassify a win as a loss.
Drift auto-reset
3 consecutive weekly JS-divergence breaches → bandits reinitialized from latest frequencies.
